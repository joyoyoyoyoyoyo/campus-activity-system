# 配置管理文档

> 规定本项目仓库、分支、提交、评审、版本与文档协作的统一约定，确保多人协作下结构清晰、变更可追溯。

## 1. 仓库与目录

- 唯一仓库：`campus-activity-system`，所有交付物统一汇入。
- 目录见根目录 `README.md` "目录结构" 一节，未经评审不得新增一级目录。

## 2. 分支策略

采用简化的"主干 + 功能分支"模式：

| 分支       | 用途                                      | 谁能写        |
| ---------- | ----------------------------------------- | ------------- |
| `main`     | 稳定可交付，所有最终成果汇总至此           | 仅经合并请求  |
| `dev`      | 集成开发分支，组员合并去向                 | 经合并请求    |
| `feat/*`   | 单个功能或工作包的开发分支                 | 任意成员      |
| `docs/*`   | 文档专用分支（WBS、报告等）                | 任意成员      |
| `fix/*`    | 缺陷修复分支                               | 任意成员      |

命名示例：

- `feat/FR-02-student-signup`
- `docs/wbs-v1`
- `fix/checkin-duplicate`

## 3. 提交（Commit）规范

格式：`<type>(<scope>): <subject>`

| type  | 含义           |
| ----- | -------------- |
| feat  | 新功能         |
| fix   | 缺陷修复       |
| docs  | 文档变更       |
| refactor | 重构        |
| test  | 测试相关       |
| chore | 构建 / 杂项     |

要求：

- 主语动词原形，subject ≤ 50 字。
- 一次提交聚焦一件事，避免大杂烩。
- 涉及需求条目时在正文引用，例如 `Refs: FR-02-02`。

## 4. 合并请求（Pull Request / Merge Request）

- 任何向 `dev` / `main` 的合并必须经 PR。
- PR 描述需包含：变更概述、关联 WBS / 需求编号、测试情况。
- 至少 1 名其他成员评审通过，方可合并。
- 禁止 force push 到 `main` / `dev`。

## 5. 版本与发布

- 采用 `vMAJOR.MINOR` 简化语义化：
  - `v0.x` 开发期；`v1.0` 项目最终交付。
- 每次里程碑（见进度计划 M1–M6）打 tag。

## 6. 文档与代码协同

- 文档与代码同仓库；不接受外发 Word 作为唯一来源。
- 需求 / WBS / 进度文档变更也必须通过 PR。
- 总结报告（`docs/summary-report.md`）由全员协作编辑，禁止单人覆盖式提交。

## 7. 配置项清单（CIs）

| 类别       | 配置项                                         | 责任人 |
| ---------- | ---------------------------------------------- | ------ |
| 文档       | `README.md`、`docs/*.md`                        | 全员   |
| 源代码     | `src/backend/`、`src/frontend/`                | 开发组 |
| 测试       | `tests/`                                       | 测试组 |
| 项目配置   | `config/`、`.gitignore`                         | 维护组 |

## 8. 变更控制流程

1. 提出变更 → 在 issue / PR 中说明动机与影响范围。
2. 组内评审 → 至少 1 人确认。
3. 合并入 `dev`，必要时回归测试。
4. 里程碑结束统一并入 `main`。

## 9. 备份与归档

- 关键里程碑（M1–M6）后，由组长打 tag 并在 issue 中记录归档说明。
- 最终交付物（M6）以 `v1.0` tag 为准，对应 `main` 分支。
